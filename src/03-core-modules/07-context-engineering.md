# 上下文工程

## 前端视角理解

上下文管理类比 **React 的 Virtual DOM diff + 增量更新**：

- 全量渲染 → 发完整上下文 → 太贵
- Virtual DOM diff → 上下文摘要 → 保留关键信息
- 增量更新 → 追加新内容
- GC 回收 → 移除不再需要的上下文

也可以类比 **浏览器对网络请求的优化**：
- HTTP 压缩（gzip）→ 上下文摘要
- 缓存策略（Cache-Control）→ 重复利用
- 分页加载（Pagination）→ 分段处理
- 垃圾回收（Memory GC）→ 淘汰旧消息

## 上下文挑战

LLM 的上下文窗口虽大（100k+ tokens），但：
- 长上下文**贵**（token = 钱）
- 长上下文**慢**（推理时间增长）
- 长上下文可能**注意力稀释**

DeerFlow 通过三层策略解决：

## 策略一：子代理上下文隔离

```
主导代理上下文（大）
├── 目标：用户需求和全局计划
├── 历史：仅保留关键决策
├── 工具：全部可用
└── 技能：所有启用技能
        ↓  任务分解
子代理A上下文（小）       子代理B上下文（小）
├── 目标：子任务A           ├── 目标：子任务B
├── 历史：仅子任务A相关      ├── 历史：仅子任务B相关
├── 工具：过滤后的工具集      ├── 工具：过滤后的工具集
└── 技能：子任务所需技能      └── 技能：子任务所需技能
```

**核心原则**：每个子代理只能看到自己需要的上下文，不能窥见主导代理或其他子代理的上下文。

## 策略二：总结

```
SummarizationMiddleware
        ↓
当上下文超过 tokens 阈值时激活（可配置）
        ↓
keep_tokens: 保留最近的 token 数
trim_tokens: 超过此值时触发摘要
        ↓
对早期消息进行总结压缩
        ↓
保持 LLM 对最新消息的全分辨率访问
```

## 策略三：严格工具调用恢复

当 LLM 提供者中断工具调用循环时（比如强制停止），DeerFlow 会：

```
1. 检测提供者的中断
        ↓
2. 剥离 raw tool-call 元数据
        ↓
3. 为悬挂的工具调用注入占位结果
        ↓
4. 下次模型调用前恢复完整的历史
        ↓
5. 防止 OpenAI 兼容推理模型因 tool_call_id 校验失败
```

**前端类比**：类似 Redux 中间件中处理取消的 API 请求——dispatch 一个 fallback action 避免状态不一致。

## 工程意义

```
无上下文工程             有上下文工程
──────────────────────────────────────
提示词逐渐膨胀           上下文稳定在合理范围
早期信息被遗忘           早期信息被摘要保留
子代理互相干扰           子代理完全隔离
长任务 OOM              长任务稳定运行
工具调用链条断裂         工具调用自动恢复
```

## 关键源码路径

```python
backend/packages/harness/deerflow/agents/
├── lead_agent/
│   ├── agent.py             ← 上下文隔离（子代理专用工厂）
│   └── system_prompt.py     ← 系统提示词注入
├── middlewares/
│   ├── summarization.py     ← 总结中间件
│   └── deferred_tool_filter.py  ← 延迟加载的工具过滤
└── subagents/
    └── executor.py          ← 子代理上下文构建
```
